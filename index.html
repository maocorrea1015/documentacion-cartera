Sistema de gestión de cartera

SaaS multiempresa REST API IA predictiva v1.0

Plataforma web para administración de cuentas por cobrar de múltiples empresas. Gestiona clientes, facturas, pagos, cartera vencida y análisis de riesgo mediante inteligencia artificial.

Frontend
Vue.js 3
Backend
Flask · Python
Base de datos
PostgreSQL 15
IA
Scikit-Learn

Arquitectura general

El frontend en Vue.js 3 consume una API REST desarrollada en Flask. El backend centraliza toda la lógica de negocio, autenticación, autorización y ejecución de modelos de IA. Todos los datos se almacenan en PostgreSQL. La infraestructura corre sobre Docker con Nginx como reverse proxy y Gunicorn como servidor WSGI.

El módulo de inteligencia artificial está disponible únicamente para empresas con plan Enterprise. El control se aplica tanto en el backend (decorador @requiere_plan) como en el frontend (composable usePlan()).

Modelo multiempresa

Cada registro del sistema lleva un campo empresa_id que lo aísla completamente. Ninguna empresa puede ver ni modificar datos de otra. El empresa_id se extrae siempre del token JWT, nunca de parámetros del cliente.

Estructura de un store Pinia

// stores/clientes.store.js
export const useClientesStore = defineStore('clientes', {
  state: () => ({
    clientes: [],
    clienteActual: null,
    loading: false,
    error: null,
    pagination: { page: 1, per_page: 20, total: 0 },
  }),
  actions: {
    async fetchClientes(params = {}) {
      this.loading = true
      try {
        const res = await ClientesService.getAll(params)
        this.clientes = res.data.items
        this.pagination = res.data.pagination
      } catch (e) {
        this.error = e.message
      } finally {
        this.loading = false
      }
    }
  }
})

Guards de navegación

router.beforeEach((to, from, next) => {
  const auth = useAuthStore()
  if (!to.meta.public && !auth.isAuthenticated) return next('/login')
  if (to.meta.roles && !to.meta.roles.includes(auth.userRole))
    return next('/dashboard')
  next()
})

Interceptor Axios — refresh automático

api.interceptors.response.use(
  res => res,
  async err => {
    if (err.response?.status === 401) {
      await useAuthStore().refresh()  // Llama POST /auth/refresh
    }
    return Promise.reject(err)
  }
)

Backend — seguridad y JWT

Backend

Tokens JWT

TokenVigenciaContenido
access_token15 minutosuser_id, empresa_id, rol, plan
refresh_token7 díasuser_id
El empresa_id se extrae siempre del JWT. Nunca aceptarlo como parámetro de la request. Todo query de base de datos lleva filter_by(empresa_id=get_empresa_id()).

Decoradores de control de acceso

@requiere_roles('admin', 'supervisor')    # Validación de rol
@requiere_plan('enterprise')              # Validación de plan comercial

# Ejemplo combinado:
@jwt_required()
@requiere_roles('admin', 'supervisor', 'analista')
@requiere_plan('enterprise')
def post(self):
    ...

Regla de aislamiento multiempresa

# CORRECTO — siempre filtrar por empresa_id del token
Cliente.query.filter_by(id=cliente_id, empresa_id=get_empresa_id()).first()

# INCORRECTO — nunca consultar sin empresa_id
Cliente.query.filter_by(id=cliente_id).first()

from argon2 import PasswordHasher

hashear_password  = lambda pwd: ph.hash(pwd)

Trigger — aplicar pago

Al insertar un registro en pagos, el trigger trg_aplicar_pago descuenta el saldo_pendiente de la factura y la marca como pagada si el saldo llega a cero. También valida que el pago no exceda el saldo.

Este trigger garantiza consistencia incluso si alguien inserta un pago directamente en la base de datos, sin pasar por la API.

Trigger — timestamps automáticos

Las tablas empresas, usuarios, clientes, facturas y acuerdos_pago tienen triggers BEFORE UPDATE que actualizan automáticamente el campo actualizada_en.

Función — marcar facturas vencidas

SELECT marcar_facturas_vencidas();

Marca como vencida toda factura en estado pendiente cuya fecha_vencimiento ya pasó. Debe ejecutarse con un cron job diario, por ejemplo con pg_cron:

-- Con pg_cron (instalar extensión)
SELECT cron.schedule('0 1 * * *', 'SELECT marcar_facturas_vencidas()');
-- Ejecuta todos los días a la 1:00 AM

Consideraciones adicionales

  • Usar NUMERIC(18,2) para todos los valores monetarios — nunca FLOAT (imprecisión)
  • Usar TIMESTAMPTZ para todas las fechas con hora — almacena en UTC
  • audit_log usa BIGSERIAL (BIGINT) porque acumulará el mayor volumen de registros
  • Evaluar particionamiento de audit_log y predicciones_ia por empresa_id cuando supere los 10M de filas
  • Configurar backups automáticos diarios con retención mínima de 30 días
